# Архитектурный проект компонента «Оптимизация маршрутов доставки» системы TransLogix

## Раздел 1. Архитектурное видение и контекст

### 1.1. Описание целей системы и архитектуры

**Бизнес-цели.** Компонент «Оптимизация маршрутов доставки» системы TransLogix нацелен на решение ключевых задач логистического оператора: сокращение времени доставки, снижение расхода топлива, уменьшение нагрузки на транспорт и повышение общей эффективности логистических операций. Система должна обеспечить автоматическое планирование маршрутов с учётом пробок, погодных условий и состояния транспортных средств, а также динамическую корректировку маршрутов в реальном времени.

**Ключевые сценарии использования:**
1. **Планирование доставки** — логист загружает список заказов с адресами и временными окнами, система рассчитывает оптимальные маршруты для описанного парка транспортных средств учитывая их грзоподёмность.
2. **Динамическая корректировка** — в процессе выполнения доставки система получает данные о пробках или изменении погоды и перестраивает маршруты курьеров.
3. **Мониторинг и аналитика** — диспетчер визуально контролирует выполнение маршрутов на карте, получает прогнозы времени прибытия и решает внештатные ситуации.

**Ожидаемые метрики:**
- Точность прогноза времени доставки: **не менее 95%**.
- Доступность системы: **не менее 99,9%** (SLA).
- Скорость вывода релизов: два раза в месяц обновления для модуля алгоритмов, ежегодные — для интерфейсной части.
- Время расчёта маршрута описывается асимптотической сложностью (n - количество точек доставки):

O(n^2) для этапа построения матрицы попарных расстояний (с учётом кэширования геоданных),

O(n·k·iter) для кластеризации заказов (k – число кластеров/машин),

O(n log n) для эвристической маршрутизации внутри кластеров (на основе алгоритмов типа LKH или OR-Tools).

При n = 100 точек время выполнения на эталонной конфигурации (4 vCPU, 16 ГБ RAM) не должно превышать 5 секунд. Для n = 500 допустимое время – порядка 2 минут (с учётом параллельной обработки кластеров). Эти оценки служат основой для горизонтального масштабирования Route Optimizer.

**Вывод по разделу.** Определеные асимтотисечкие ожидания сложности системы. Особенно важным оказалось, что система должна работать в двух режимах — плановом (пакетная оптимизация) и оперативном (динамическая корректировка), что предъявляет разные требования к архитектуре и требует разделения этих потоков обработки.

### 1.2. Архитектурные драйверы

На основе требований технического задания выделены следующие нефункциональные требования с их приоритизацией:

| Приоритет | Требование | Обоснование |
|-----------|------------|-------------|
| 1 | **Масштабируемость** | Система должна обрабатывать растущее число заказов и транспортных средств; микросервисная архитектура позволяет горизонтально масштабировать вычислительно-ёмкие компоненты |
| 1 | **Надёжность и отказоустойчивость** | Доставка — критичный бизнес-процесс; недоступность системы ведёт к срыву доставок и финансовым потерям |
| 2 | **Производительность** | Первоначальные оптимальный расчёт всех маршрутов должен происходить за ночь с запасом |
| 2 | **Интегрируемость** | Система должна взаимодействовать с картографическими сервисами, погодными API, GPS-сервисами и внутренними модулями TransLogix |
| 3 | **Безопасность** | API-ключи к внешним сервисам, данные о заказах и транспорте требуют защиты |
| 3 | **Наблюдаемость** | Для выполнения SLA 99,9% необходима развитая система мониторинга и трассировки |

**Вывод по разделу.** Приоритизация нефункциональных требований показала, что масштабируемость и надёжность являются критическими драйверами, определяющими выбор микросервисной архитектуры и паттернов отказоустойчивости. Интересным открытием стало то, что производительность возможно диктует необходимость отдельного вычислительного сервиса с возможностью горизонтального масштабирования, а не просто оптимизации кода.

- Модуль транспорта — предоставляет данные о ТС (грузоподъёмность, состояние).

**Основные потоки данных:**
1. Запрос на оптимизацию > получение данных о заказах и ТС > получение статисеских данных с картографических сервисов > расчёт маршрутов > возврат оптимизированных маршрутов.
2. В процессе доставки — получение данных о пробках/погоде > перерасчёт > отправка обновлённых маршрутов.
3. Формирование отчётности по эффективности маршрутов и затратам.

**Вывод по разделу.** Контекстная диаграмма наглядно показала, что компонент не является изолированным — он находится в центре сложной экосистемы внутренних и внешних систем. Это потребовало выделить модуль интеграции как отдельную архитектурную единицу, ответственную за трансформацию данных и управление API-ключами.

## Раздел 2. Логическая архитектура и доменное моделирование

### 2.1. Выделение доменных областей и bounded contexts

На основе анализа бизнес-процессов и требований ТЗ выделены следующие доменные области и ограниченные контексты (bounded contexts):

| Домен | Bounded Context | Ответственность |
|-------|-----------------|-----------------|
| **Маршрутизация** | Route Optimization | Расчёт оптимальных маршрутов, применение алгоритмов графов и ML, прогнозирование времени доставки |
| **Автопарк и персонал** | Fleet Data | Хранение и обработка данных о транспортных средствах (грузоподъёмность, габариты, состояние), водителях (графики работы, компетенции/категории прав, закрепление за ТС) |
| **Геоданные** | Geospatial Data | Хранение и обработка картографических данных, геокодирование, расчёт расстояний |
| **Интеграция** | Integration Gateway | Взаимодействие с внешними API (карты, погода, GPS), трансформация данных, управление API-ключами |
| **Мониторинг доставки** | Delivery Tracking | Отслеживание выполнения маршрутов, обработка GPS-данных, динамическая корректировка |
| **Аналитика** | Analytics | Формирование отчётов по эффективности маршрутов, времени доставки, затратам |
| **Управление настройками** | Configuration Management | Управление параметрами оптимизации, приоритетами, настройками внешних сервисов |

Каждый контекст имеет свой универсальный язык (ubiquitous language) и собственную модель данных. Границы контекстов проведены по критериям:
- **Изменяемость** — алгоритмы оптимизации меняются чаще, чем геоданные.
- **Масштабируемость** — вычислительные задачи требуют отдельного масштабирования.
- **Внешние зависимости** — интеграция с внешними API изолирована в отдельный контекст.

**Вывод по разделу.** Выделение bounded contexts через призму DDD позволило увидеть систему не как монолитную задачу «оптимизации маршрутов», а как совокупность взаимодействующих доменов с разной динамикой изменений. Наиболее ценным оказалось осознание, что контексты «Route Optimization» и «Delivery Tracking» требуют принципиально разных архитектурных подходов — первый ориентирован на пакетную обработку, второй — на потоковую в реальном времени.

### 2.2. Доменные модели внутри ключевых контекстов

#### Контекст «Route Optimization»

**Агрегаты:**
- **OptimizationTask** — корневой агрегат, представляющий задачу оптимизации. Содержит: список заказов, парк ТС, параметры оптимизации, статус, результат.
- **Route** — маршрут для одного ТС. Содержит: идентификатор ТС, последовательность точек доставки, прогнозируемое время, расстояние, стоимость.

**Сущности:**
- **DeliveryPoint** — точка доставки с адресом, временным окном, приоритетом.
- **Vehicle** — транспортное средство (получается из внешнего модуля, но в контексте имеет проекцию с необходимыми атрибутами).

**Value Objects:**
- **Coordinates** — географические координаты (широта, долгота).
- **TimeWindow** — временной интервал для доставки.
- **RouteMetrics** — метрики маршрута (расстояние, время, стоимость топлива).

**Инварианты:**
- Суммарная загрузка ТС не должна превышать грузоподъёмность.
- Время прибытия в каждую точку должно укладываться в заданное временное окно.
- Одно ТС не может находиться в двух точках одновременно.

#### Контекст «Delivery Tracking»

**Агрегаты:**
- **DeliveryExecution** — выполнение доставки. Содержит: ссылку на маршрут, текущий статус, историю позиций.
- **Position** — позиция ТС с временной меткой и координатами.

**Value Objects:**
- **GeoPosition** — координаты с временем фиксации.
- **Deviation** — отклонение от планового маршрута.

**Инварианты:**
- Позиция ТС должна обновляться не реже раза в минуту.
- При отклонении более чем на 500 метров от маршрута должна генерироваться предупреждение.

**Вывод по разделу.** Построение доменных моделей позволило формализовать бизнес-правила и инварианты, которые ранее существовали только на уровне требований. Особенно важным оказалось выделение Value Objects — Coordinates, TimeWindow, RouteMetrics — которые не имеют идентичности и могут свободно передаваться между сервисами. Это упрощает межсервисное взаимодействие и снижает связанность.

### 2.3. Междоменные взаимодействия

**Доменные события:**

| Событие | Источник | Назначение | Описание |
|---------|----------|------------|----------|
| `OptimizationRequested` | UI / Диспетчер | Route Optimization | Запрос на расчёт маршрутов |
| `RoutesOptimized` | Route Optimization | Delivery Tracking, Analytics | Маршруты рассчитаны и готовы к выполнению |
| `RouteUpdated` | Delivery Tracking | Route Optimization | Маршрут скорректирован в реальном времени |
| `DeliveryStarted` | Delivery Tracking | Analytics | Начало выполнения доставки |
| `DeliveryCompleted` | Delivery Tracking | Analytics | Завершение доставки |
| `DeviationDetected` | Delivery Tracking | Route Optimization | Обнаружено отклонение от маршрута |
| `ExternalDataUpdated` | Integration Gateway | Route Optimization | Обновлены данные о пробках/погоде |

**Ключевой бизнес-процесс: «Планирование доставки» через события**

```
1. Логист инициирует планирование > UI публикует OptimizationRequested
2. Route Optimization получает событие, запрашивает через Integration Gateway:
   - Данные о заказах (из модуля заказов)
   - Данные о ТС (из модуля транспорта)
   - Данные о пробках и погоде (из внешних API)
3. Route Optimization выполняет расчёт > публикует RoutesOptimized
4. Delivery Tracking получает маршруты > готовится к мониторингу
5. Analytics получает маршруты > фиксирует плановые показатели
```

**Ключевой бизнес-процесс: «Динамическая корректировка маршрута»**

```
1. Integration Gateway получает обновление о пробках > публикует ExternalDataUpdated
2. Route Optimization пересчитывает затронутые маршруты > публикует RouteUpdated
3. Delivery Tracking обновляет маршруты для курьеров > уведомляет через UI
4. Analytics фиксирует отклонение и пересчёт
```

**Вывод по разделу.** Описание процессов через события позволило увидеть, что система по своей природе является событийно-ориентированной. Это естественным образом привело к выбору асинхронных взаимодействий через брокер сообщений для большинства сценариев. Наиболее ценным было осознание, что синхронные вызовы (через REST API) должны использоваться только для запросов, требующих немедленного ответа (например, получение данных о заказах), а все процессы, связанные с изменением состояния, должны быть асинхронными.

## Раздел 3. Архитектура микросервисов

### 3.1. Состав микросервисов

| Сервис | Домен | Ответственность | Основные операции | Владеемые данные |
|--------|-------|-----------------|-------------------|------------------|
| **Route Optimizer** | Route Optimization | Расчёт оптимальных маршрутов с использованием алгоритмов A*, Dijkstra, ML (регрессия, кластеризация) | `optimize()`, `reoptimize()`, `predictETA()` | Кэш маршрутов, ML-модели, временные данные расчётов |
| **Fleet Management** | Fleet Data | Хранение и предоставление актуальных данных о транспортных средствах (грузоподъёмность, габариты, состояние), водителях (графики работы, компетенции/категории прав, закрепление за ТС) | `getAvailableVehicles()`, `getVehicleById()`, `getDriversSchedule()`, `validateDriverCompliance()` | PostgreSQL — ТС (модель, госномер, грузоподъёмность, габариты, состояние), водители (ФИО, график, категории прав, контакты) |
| **Geospatial Service** | Geospatial Data | Хранение и обработка геоданных, геокодирование, расчёт матриц расстояний | `geocode()`, `calculateDistance()`, `getRoadNetwork()` | PostGIS — картографические данные, геокоды, расстояния |
| **Integration Gateway** | Integration Gateway | Взаимодействие с внешними API (карты, погода, GPS), трансформация данных, управление API-ключами | `fetchTraffic()`, `fetchWeather()`, `fetchGPS()` | API-ключи, конфигурации внешних сервисов, кэш внешних данных |
| **Delivery Tracker** | Delivery Tracking | Отслеживание выполнения маршрутов в реальном времени, обработка GPS-данных | `trackDelivery()`, `detectDeviation()`, `updatePosition()` | Текущие позиции ТС, статусы доставок, история трекинга |
| **Analytics Service** | Analytics | Формирование отчётов по эффективности маршрутов, времени доставки, затратам | `generateReport()`, `getMetrics()`, `calculateSavings()` | Агрегированные метрики, исторические отчёты |
| **Config Service** | Configuration Management | Управление параметрами оптимизации и настройками внешних сервисов | `getConfig()`, `updateConfig()`, `validateConfig()` | Параметры оптимизации, настройки интеграций |
| **API Gateway** | Cross-cutting | Маршрутизация запросов, аутентификация, rate limiting, агрегация ответов | `route()`, `authenticate()`, `rateLimit()` | Не хранит данные (stateless) |

**Вывод по разделу.** Состав микросервисов определён на основе выделенных bounded contexts, что обеспечивает согласованность между логической и физической архитектурами. Ключевым решением стало выделение Integration Gateway как отдельного сервиса — это позволяет централизованно управлять всеми внешними интеграциями, упрощает замену картографических провайдеров и изолирует остальные сервисы от изменений во внешних API.

Использование Kafka как событийной шины обеспечивает гарантированную доставку сообщений и позволяет легко добавлять новых потребителей событий.

### 3.3. Анализ границ сервисов

**Обоснование выбора границ:**

Границы сервисов определены на основе трёх принципов:

1. **Бизнес-возможности (Business Capabilities)** — каждый сервис соответствует отдельной бизнес-способности: оптимизация, трекинг, аналитика, интеграция.

2. **Частота изменений** — Route Optimizer меняется часто (эволюция алгоритмов), Geospatial Service — редко (стабильные картографические данные). Разделение позволяет обновлять алгоритмы без переразвёртывания геосервиса.

3. **Масштабирование** — Route Optimizer требует горизонтального масштабирования под нагрузку, в то время как Config Service и Geospatial Service масштабируются иначе.

**Альтернативы и риски:**

| Альтернатива | Преимущества | Риски |
|--------------|--------------|-------|
| **Объединение Integration Gateway и Geospatial Service** | Меньше сетевых вызовов, проще транзакции | Проблема логики Integration Gateway не должно заниматься какими либо расчётами |
| **Разделение Route Optimizer на Planning и Reoptimization** | Более тонкое масштабирование | Дополнительная сложность координации; избыточность для текущих объёмов |
| **Монолит (один сервис)** | Проще разработка и деплой | Невозможность независимого масштабирования; единая точка отказа |

**Выбранное решение:** 7 сервисов с чёткими границами ответственности. Это обеспечивает:
- Независимое масштабирование Route Optimizer при пиковых нагрузках.
- Изоляцию изменений — обновление алгоритмов не влияет на трекинг и аналитику.
- Возможность замены картографического провайдера через Integration Gateway без изменения других сервисов.

**Риски выбранного решения:**
- Увеличение сетевого трафика между сервисами.
- Сложность обеспечения согласованности данных между сервисами (решается через Saga и событийную модель).
- Дополнительные накладные расходы на оркестрацию (Kubernetes).

**Вывод по разделу.** Анализ границ сервисов показал, что выбор количества сервисов — это всегда компромисс между гибкостью и сложностью. Семь сервисов представляются оптимальным числом для TransLogix: достаточно для независимого масштабирования и изоляции изменений, но не настолько много, чтобы управление становилось неподъёмным. Ключевым критерием при определении границ стала частота изменений каждого компонента.

## Раздел 4. Архитектура данных и согласованности

### 4.1. Модель владения данными

В соответствии с паттерном «Database per Service», каждый микросервис владеет своей базой данных и не имеет прямого доступа к данным других сервисов:

| Сервис | Владеемые сущности | Тип хранилища | Аргументация |
|--------|-------------------|---------------|--------------|
| **Route Optimizer** | OptimizationTask, Route (кэш), ML-модели | **PostgreSQL + Redis** | PostgreSQL — для персистентного хранения задач и результатов; Redis — для кэширования маршрутов и быстрого доступа к ML-моделям |
| **Fleet Management** | Данных о транспортных средствах, водителях и возможно скалдах |  PostgreSQL — обычная БД |
| **Geospatial Service** | Картографические данные, геокоды, матрицы расстояний | **PostgreSQL + PostGIS** | PostGIS обеспечивает эффективную работу с пространственными данными и индексацию географических запросов |
| **Integration Gateway** | API-ключи, конфигурации внешних сервисов, кэш внешних данных | **PostgreSQL + Redis** | PostgreSQL — для конфигураций и ключей; Redis — для кэширования ответов внешних API (TTL 30 дней для геокодирования) |
| **Delivery Tracker** | DeliveryExecution, Position, история трекинга | **TimescaleDB (PostgreSQL)** | TimescaleDB оптимизирована для временных рядов — идеально для хранения GPS-позиций с временными метками |
| **Analytics Service** | Агрегированные метрики, исторические отчёты | **PostgreSQL + ClickHouse** | PostgreSQL — для оперативных отчётов; ClickHouse — для OLAP-запросов по историческим данным |
| **Config Service** | Параметры оптимизации, настройки | **PostgreSQL** | Стандартное реляционное хранилище для конфигураций |

**Выбор PostgreSQL как основного хранилища** обоснован требованиями ТЗ (п. 4, Технические требования), а также возможностью гибридного использования — документы в JSONB и реляционные данные в одной СУБД.

**Вывод по разделу.** Применение паттерна «Database per Service» обеспечивает слабую связанность сервисов и позволяет выбирать оптимальный тип хранилища для каждой задачи. Наиболее интересным решением стало использование TimescaleDB для Delivery Tracker — осознание того, что GPS-данные являются временными рядами, а не просто реляционными данными, позволило выбрать специализированное хранилище, которое обеспечит высокую производительность при минимальных затратах на разработку.

### 4.2. Межсервисная согласованность

Для обеспечения согласованности данных между сервисами в распределённой системе используется комбинация паттернов.

#### Паттерн Saga (оркестрация) для процесса «Планирование доставки»

Шаг 1: Получить данные о заказах (модуль заказов TransLogix)        
* Успех: продолжаем 
* Ошибка: компенсация — отмена задачи    
                                                                    
Шаг 2: Получить данные о ТС (модуль транспорта TransLogix)          
* Успех: продолжаем 
* Ошибка: компенсация — отмена задачи    
                                                                    
Шаг 3: Получить данные о пробках (Integration Gateway > внешний API)
* Успех: продолжаем
* Ошибка: компенсация — использовать кэш 
                                                                    
Шаг 4: Выполнить оптимизацию (внутренний расчёт)                    
* Успех: публикуем RoutesOptimized 
* Ошибка: компенсация — логируем ошибку, уведомляем пользователя                   
                                                                    
Шаг 5: Сохранить результат (в свою БД)                              
* Успех: завершаем Saga 
* Ошибка: компенсация — откат        

**Компенсирующие действия:**
- При ошибке на любом шаге — задача оптимизации переводится в статус `FAILED`, пользователь получает уведомление.
- Если ошибка произошла после публикации `RoutesOptimized` — Delivery Tracker и Analytics получают событие с пометкой о необходимости игнорирования (или компенсации).

#### Паттерн «Eventually Consistent» для процесса «Динамическая корректировка»

При обновлении маршрута в реальном времени допускается временная несогласованность:

1. Route Optimizer пересчитывает маршрут и публикует `RouteUpdated`.
2. Delivery Tracker обновляет свой кэш асинхронно (задержка < 1 секунда).
3. Analytics получает событие для обновления метрик (задержка до 5 секунд).
4. Обновление одно и того же маршрута запрещено проводить чаще, чем раз в 3 минуты.

Это допустимо, поскольку:
- Курьер получает обновлённый маршрут через WebSocket (синхронно).
- Аналитические данные не критичны к задержке в несколько секунд.

**Вывод по разделу.** Выбор паттернов согласованности показал, что не все процессы требуют строгой согласованности. Для «Планирования доставки» необходима Saga с компенсациями, поскольку это критичный процесс, где потеря данных недопустима. Для «Динамической корректировки» достаточно eventual consistency — это позволило упростить архитектуру и повысить производительность. Ключевым критерием выбора стала критичность процесса для бизнеса.

### 4.3. Стратегии эволюции и миграции данных

#### Эволюция схем данных

Для управления изменениями схем данных применяются следующие стратегии:

**1. Добавление новых полей (совместимое изменение):**
- Новые поля добавляются с значениями по умолчанию.
- Старые версии сервисов игнорируют новые поля (tolerant reader).
- Пример: добавление `priority` в `DeliveryPoint` — старые версии используют `priority = MEDIUM` по умолчанию.

**2. Удаление или переименование полей (ломающее изменение):**
- Выполняется в несколько этапов:
  1. Добавить новое поле, пометить старое как `@Deprecated`.
  2. Обновить все потребители на использование нового поля.
  3. Удалить старое поле через 2 релизных цикла.

**3. Миграция типов данных:**
- Используется паттерн «expand-contract»: добавить новое поле, мигрировать данные, удалить старое.

#### Миграция из монолита (применимо)

Если существующая система TransLogix имеет монолитную архитектуру, миграция компонента выполняется по паттерну «Strangler Fig»:

Этап 1: Anti-Corruption Layer
* Создать ACL (Anti-Corruption Layer) между монолитом и новым компонентом. 
* Монолит продолжает работать, все запросы к оптимизации проходят через ACL.
* Сравнивается точность резульаттов предсказаний для монолитной и не монолитнйо системы.

Этап 2: Постепенное переключение трафика
* 10% запросов > новый компонент, 90% > монолит.
* Постепенно увеличивать долю до 100% при успешных результатах мониторинга.
                                Ў
Этап 3: Завершение миграции
* Весь трафик на новый компонент. 
* Монолитный код оптимизации выводится из эксплуатации.

**Вывод по разделу.** Стратегии эволюции данных и миграции из монолита показали, что архитектурное проектирование должно учитывать не только «зелёное поле», но и путь от существующего состояния. Паттерн Strangler Fig оказался особенно ценным — он позволяет мигрировать постепенно, снижая риски и давая возможность отката на каждом этапе. Принцип «tolerant reader» (игнорирование неизвестных полей) стал ключевым для обеспечения обратной совместимости.

## Раздел 5. Архитектура API и интеграций

### 5.1. Внешний API системы

**Роль API Gateway:**

API Gateway является единой точкой входа для всех внешних запросов к системе. Он выполняет следующие функции:
- **Маршрутизация** — направление запросов к соответствующим микросервисам.
- **Аутентификация и авторизация** — проверка JWT-токенов и прав доступа.
- **Rate Limiting** — ограничение числа запросов от одного клиента для предотвращения перегрузки.
- **Агрегация ответов** — объединение данных из нескольких сервисов для сложных запросов.
- **Версионирование API** — управление версиями публичных API.

**Основные публичные API:**

| Эндпоинт | Метод | Описание | Версионирование |
|----------|-------|----------|-----------------|
| `/api/v1/optimize` | POST | Запуск оптимизации маршрутов | URL-based (/v1/, /v2/) |
| `/api/v1/optimize/{taskId}/status` | GET | Получение статуса задачи | URL-based |
| `/api/v1/optimize/{taskId}/routes` | GET | Получение результатов оптимизации | URL-based |
| `/api/v1/tracking/{deliveryId}/position` | GET | Получение текущей позиции ТС | URL-based |
| `/api/v1/analytics/reports` | GET | Получение отчётов по эффективности | URL-based |
| `/api/v1/config` | GET/PUT | Управление настройками оптимизации | URL-based |
| `/api/v1/fleet` | GET/PUT | Управление списком доступного трансопрта и персонала | URL-based |

**Принципы версионирования**:
- URL-based версионирование: `/api/v1/...`, `/api/v2/...`
- Обратная совместимость в течение 12 месяцев после выхода новой версии.
- Deprecation-уведомления через HTTP-заголовок `Deprecation: true`.
- Аддитивные изменения не требуют создания новой версии.

**Вывод по разделу.** API Gateway решает множество кросс-сервисных задач в одном месте, что упрощает каждый отдельный микросервис. Наиболее важным решением стало использование URL-based версионирования — оно наиболее прозрачно для клиентов и легко реализуется на уровне Gateway. Отказ от версионирования аддитивных изменений (следуя принципу «не версионируйте то, что можно не версионировать») позволит снизить эксплуатационную нагрузку.

### 5.3. Интеграции с внешними системами

| Внешняя система | Способ интеграции | Anti-Corruption Layer |
|-----------------|-------------------|----------------------|
| Google Maps API | REST (синхронно) | Transform ответов в единый внутренний формат; кэширование результатов |
| Яндекс.Карты API | REST (синхронно) | Transform ответов в единый внутренний формат; fallback при недоступности Google |
| Погодные API | REST (синхронно) с кэшированием | Кэширование на 1 час; fallback до последнего успешного ответа |
| GPS-сервисы | WebSocket / MQTT (асинхронно) | Преобразование потоков GPS-данных в события для Kafka |
| Модуль заказов TransLogix | REST (синхронно) | ACL для трансформации формата данных |
| Модуль транспорта TransLogix | REST (синхронно) | ACL для трансформации формата данных |

**Anti-Corruption Layer (ACL)**:
- Каждая внешняя интеграция проходит через ACL, который трансформирует внешние модели данных во внутренние.
- ACL изолирует остальную систему от изменений во внешних API.
- При смене картографического провайдера меняется только ACL, остальные сервисы не затрагиваются.

**Стратегия fallback для картографических сервисов:**
1. Основной провайдер: Google Maps API.
2. Резервный: Яндекс.Карты API (при недоступности Google).
3. Кэшированные данные (при недоступности обоих).

**Управление API-ключами:**
- Ключи хранятся в Config Service с шифрованием (AWS KMS / HashiCorp Vault).
- Ротация ключей выполняется автоматически каждые 90 дней.
- Доступ к ключам имеют только Integration Gateway (через Config Service).

**Вывод по разделу.** Анализ внешних интеграций показал, что наиболее уязвимым местом системы являются внешние API — они могут быть недоступны, менять свои форматы или вводить rate limiting. ACL и стратегии fallback являются критическими для обеспечения отказоустойчивости. Наиболее ценным решением стало централизованное хранение API-ключей в Config Service с шифрованием — это позволяет управлять доступом к внешним сервисам в одном месте и упрощает ротацию ключей.

## Раздел 6. Качественные атрибуты и кросс-сервисные решения

### 6.1. Масштабируемость и производительность

**Стратегии масштабирования по сервисам:**

| Сервис | Способ масштабирования | Обоснование |
|--------|----------------------|-------------|
| **Route Optimizer** | Горизонтальное (Kubernetes HPA) | Вычислительная нагрузка растёт с числом заказов; масштабируется по CPU и длине очереди |
| **Integration Gateway** | Горизонтальное | Зависит от числа внешних запросов; кэширование снижает нагрузку |
| **Delivery Tracker** | Горизонтальное + шардирование | GPS-данные приходят непрерывно; шардирование по `deliveryId` |
| **Geospatial Service** | Вертикальное + read replicas | Данные стабильны, но много чтений; реплики для распределения нагрузки |
| **Analytics Service** | Отдельно для OLTP и OLAP | Оперативные запросы — на PostgreSQL; аналитические — на ClickHouse |
| **API Gateway** | Горизонтальное (stateless) | Легко масштабируется, не хранит состояние |

**Предполагаемые узкие места и способы устранения:**

| Узкое место | Причина | Решение |
|-------------|---------|---------|
| **Расчёт маршрутов** | Алгоритмическая сложность | Кэширование матриц расстояний в Redis; использование OR-Tools с оптимизацией; распараллеливание расчётов |
| **Внешние API** | Ограничения по rate limit | Кэширование (30 дней для геокодирования); batch-запросы; fallback-провайдеры |
| **Запись GPS-данных** | Высокая частота (1 запись/мин x 1000 ТС) | TimescaleDB с автоматическим партиционированием; batch-запись через Kafka |
| **Событийная шина** | Большой объём событий | Kafka с партиционированием по ключам; настройка retention под нагрузку |

**Вывод по разделу.** Масштабируемость — это не только горизонтальное добавление реплик, но и правильный выбор архитектурных паттернов для каждого сервиса. Наиболее интересным стало осознание того, что Route Optimizer требует двухуровневого масштабирования: горизонтального (больше инстансов) и алгоритмического (оптимизация самих расчётов). Кэширование матриц расстояний в Redis оказалось ключевым решением для достижения необходимой скорости расчётов.

### 6.2. Надёжность и отказоустойчивость

**Используемые паттерны**:

| Паттерн | Применение | Параметры |
|---------|------------|-----------|
| **Timeout** | Все внешние вызовы (REST, API) | Google Maps: 3 сек; Погодные API: 2 сек; Внутренние сервисы: 5 сек |
| **Retry** | Временные сбои (сеть, таймауты) | Макс. 3 попытки; exponential backoff: 1с, 2с, 4с |
| **Circuit Breaker** | Защита от каскадных отказов | Порог: 5 ошибок за 30 сек; таймаут полуоткрытого состояния: 60 сек |
| **Fallback** | При недоступности внешних сервисов | Кэшированные данные; упрощённый алгоритм (только расстояние, без учёта пробок) |
| **Bulkhead** | Изоляция нагрузки между сервисами | Отдельные пулы соединений для разных внешних API |
| **Health Checks** | Мониторинг состояния сервисов | Liveness + Readiness пробы в Kubernetes |

**Стратегии деградации:**

| Сценарий | Действие |
|----------|----------|
| Google Maps недоступен | Переключиться на Яндекс.Карты; использовать кэшированные матрицы расстояний |
| Погодное API недоступно | Использовать последние полученные данные (не старше 1 часа) |
| Route Optimizer перегружен | Приоритизация новых задач перед перерасчётом; уведомление диспетчера |
| Kafka недоступна | Сохранение событий в локальном журнале; репликация при восстановлении |

**SLA и SLO:**
- Доступность: **99,9%** (менее 43 минут простоя в месяц).
- Время восстановления (RTO): **15 минут**.
- Целевая точка восстановления (RPO): **5 минут** (потеря данных не более 5 минут).

**Вывод по разделу.** Надёжность в распределённой системе обеспечивается не одним паттерном, а их комбинацией. Наиболее важным оказалось сочетание Circuit Breaker и Fallback — это позволяет системе продолжать работу даже при отказе критических внешних сервисов, пусть и с ухудшенными характеристиками. Определение RTO и RPO дало чёткие цели для инфраструктурной команды и позволило обосновать инвестиции в резервирование.

### 6.3. Безопасность

**Модель аутентификации и авторизации:**                                                                     

1. Аутентификация: JWT (JSON Web Token) с OAuth 2.0 / OpenID Connect 
   L-- API Gateway проверяет JWT, извлекает роли и права             
                                                                     
2. Авторизация: RBAC (Role-Based Access Control)                   
   |   Роль      |                   Права                        |  
   |-------------|------------------------------------------------|
   |   Логист    |  Запуск оптимизации, просмотр всех маршрутов   |
   |   Диспетчер |  Просмотр и корректировка маршрутов в реальном времени, управление ТС |
   |   Курьер    |  Просмотр своего маршрута, отправка GPS        |
   |Администратор| Управление настройками, API-ключами, пользователями|
                                                                       
3. Защита каналов:                                                     
   L-- Все внешние коммуникации — TLS 1.3                              
   L-- Внутренние коммуникации — mTLS (service-to-service)             
   L-- Шифрование данных в покое (encryption at rest)                  
                                                                       
4. Управление секретами:                                               
   L-- API-ключи, пароли — HashiCorp Vault / AWS KMS                   
   L-- Доступ только через Config Service                              

**Разделение прав на уровне данных:**
- Логист видит все маршруты, но не может изменять настройки интеграций.
- Диспетчер может корректировать маршруты, но не может изменять алгоритмы оптимизации.
- Курьер видит только свои маршруты и точки доставки.

**Аудит и логирование безопасности:**
- Все операции аутентификации и авторизации логируются.
- Доступ к API-ключам и конфиденциальным данным аудируется.
- Логи безопасности хранятся 90 дней в отдельном защищённом хранилище.

**Вывод по разделу.** Безопасность должна быть встроена в архитектуру с самого начала, а не добавлена позже. Наиболее важным решением стало разделение аутентификации (на уровне API Gateway) и авторизации (на уровне каждого сервиса) — это позволяет централизованно управлять доступом на входе, но сохраняет гибкость в контроле прав на уровне бизнес-логики. Использование mTLS для внутренних коммуникаций обеспечивает защиту от внутренних угроз.

### 6.4. Наблюдаемость

**Три столпа наблюдаемости**:

| Сигнал | Инструменты | Что собираем | Для чего |
|--------|-------------|--------------|----------|
| **Логи** | ELK Stack (Elasticsearch, Logstash, Kibana) / Loki | Структурированные логи (JSON) с correlation ID; все запросы, ошибки, предупреждения | Отладка, аудит, анализ инцидентов |
| **Метрики** | Prometheus + Grafana | CPU, память, latency запросов, ошибки, длина очереди, количество задач | Мониторинг SLA, алертинг, capacity planning |
| **Трассировка** | Jaeger / Zipkin (OpenTelemetry) | Распределённые трейсы всех запросов через сервисы | Выявление узких мест, анализ задержек, оптимизация |

**Ключевые метрики для алертинга:**

| Метрика | Порог | Действие |
|---------|-------|----------|
| Время расчёта маршрута | > 5 сек | Предупреждение > масштабирование |
| Доступность API Gateway | < 99.9% | Критический алерт |
| Ошибки внешних API | > 5% за 5 минут | Предупреждение > проверка провайдера |
| Длина очереди Kafka | > 10000 сообщений | Предупреждение > масштабирование потребителей |
| Задержка обработки GPS | > 2 сек | Предупреждение |

**Correlation ID:**
- Каждый запрос получает уникальный `X-Request-ID` на уровне API Gateway.
- ID передаётся во все последующие вызовы и логируется.
- Позволяет проследить путь запроса через все сервисы.

**Дашборды:**
- **Operational Dashboard** — статус всех сервисов, ошибки, latency.
- **Business Dashboard** — количество оптимизаций, среднее время доставки, эффективность маршрутов.
- **Security Dashboard** — попытки несанкционированного доступа, ротация ключей.

**Вывод по разделу.** Наблюдаемость — это не просто «хорошо бы иметь», а критическое требование для выполнения SLA 99,9%. Наиболее важным решением стало внедрение correlation ID на уровне API Gateway — это позволяет связать логи, метрики и трейсы в единую картину для каждого запроса. Три столпа наблюдаемости (логи, метрики, трассировка) дополняют друг друга и дают полную картину состояния системы.

### 7.1. Целевая платформа развертывания

**Выбор платформы: облачная инфраструктура (AWS / Azure) с Kubernetes**

| Компонент | Выбор | Обоснование |
|-----------|-------|-------------|
| **Платформа** | AWS / Azure (облако) | Масштабируемость, управляемые сервисы, соответствие требованиям ТЗ |
| **Контейнеризация** | Docker | Единый формат упаковки, переносимость между средами |
| **Оркестрация** | Kubernetes (EKS / AKS) | Автоматическое масштабирование, self-healing, rolling updates |
| **Service Mesh** | Istio / Linkerd (опционально) | Управление трафиком, mTLS, наблюдаемость |

**Обоснование выбора Kubernetes**:
- **Автоматическое масштабирование** — Horizontal Pod Autoscaler для Route Optimizer.
- **Self-healing** — автоматический перезапуск упавших подов.
- **Rolling updates** — безопасные обновления без простоя.
- **Единая платформа** — Dev, Test, Stage, Prod — одинаковые среды.

**Управляемые сервисы облачного провайдера:**
- **Управляемый Kubernetes** — EKS (AWS) или AKS (Azure) — снижает операционную нагрузку.
- **Управляемая PostgreSQL** — RDS / Azure Database — автоматические бэкапы, реплики.
- **Управляемый Kafka** — MSK / Confluent Cloud — гарантированная доступность брокеров.
- **Управляемый Redis** — ElastiCache / Azure Cache — управление кэшированием.

**Вывод по разделу.** Выбор облачной платформы с Kubernetes обусловлен требованием масштабируемости и управляемости. Наиболее важным решением стало использование управляемых сервисов (RDS, MSK, ElastiCache) вместо самостоятельного развёртывания — это позволяет сосредоточиться на разработке приложения, а не на эксплуатации инфраструктуры. Kubernetes даёт единый интерфейс для всех сред и обеспечивает плавный переход от разработки к production.

### 7.2. Архитектура окружений

В системе выделяются четыре стандартных окружения, каждое из которых решает свои задачи и имеет свои особенности конфигурации. Все окружения развёрнуты в едином кластере Kubernetes, но изолированы друг от друга на уровне namespace, что обеспечивает общую инфраструктурную однородность и упрощает управление.

1. **Dev (Разработка)**
   - Предназначено для повседневной работы разработчиков.
   - Используется для локальной отладки, быстрого тестирования новых идей и проверки небольших изменений.
   - Данные – синтетические (генерируются скриптами) или анонимизированные снимки из тестового окружения.
   - Внешние сервисы (картографические API, погодные API) заменяются моками, чтобы не расходовать реальные квоты и не зависеть от сети.
   - Деплой выполняется автоматически при каждом push в ветку `develop`.

2. **Test (Автоматизированное тестирование)**
   - Окружение для прогона всех видов автоматических тестов: модульных, интеграционных, API-тестов и контрактных.
   - Здесь проверяется корректность взаимодействия между сервисами и с внешними зависимостями (внешние API уже не мокаются, но используются тестовые ключи и песочницы).
   - Данные – фиксированный набор тестовых сценариев, который позволяет воспроизводить ожидаемые результаты.
   - Деплой также автоматический, после успешного прохождения всех Quality Gates в CI.

3. **Stage (Предпроизводственное окружение)**
   - Максимально приближено к production по конфигурации, версиям ПО и используемым данным.
   - Предназначено для финального тестирования перед релизом: нагрузочные тесты, UAT (приёмочное тестирование пользователями), проверка миграций данных и сценариев восстановления.
   - Данные – регулярно обновляемая копия production-данных (анонимизированная, но сохраняющая структуру и объём).
   - Внешние интеграции работают с реальными ключами (отдельные от prod), чтобы проверить поведение системы на реальных ответах внешних API.
   - Деплой в Stage выполняется вручную (по кнопке) после утверждения тимлидом и продакт-оунером.

4. **Prod (Production)**
   - Боевое окружение, в котором обслуживаются реальные пользователи и обрабатываются реальные заказы.
   - Обеспечивает требуемый уровень доступности 99,9% (SLA).
   - Развернуто в мульти-зональной конфигурации (Multi-AZ) для отказоустойчивости.
   - Включено автоматическое горизонтальное масштабирование (HPA) под нагрузкой.
   - Деплой осуществляется по стратегии **Canary** или **Blue-Green** с автоматическим откатом при превышении порога ошибок.

**Принципы изоляции и повторяемости:**

| Принцип | Реализация |
|---------|------------|
| **Изоляция окружений** | Разные namespace в Kubernetes; отдельные экземпляры баз данных (PostgreSQL, TimescaleDB, ClickHouse), отдельные кластеры Kafka и Redis для каждого окружения. |
| **Повторяемость** | Вся инфраструктура описывается как код (Terraform / Pulumi), а конфигурации приложений – через Helm-чарты. Всё хранится в репозитории и проходит код-ревью. |
| **Управление секретами** | Секреты (API-ключи, пароли, токены) хранятся в HashiCorp Vault и подтягиваются в поды через External Secrets Operator. Для каждого окружения свой набор секретов. |
| **Данные** | Dev и Test используют синтетические / фиксированные наборы; Stage – анонимизированную копию Prod, обновляемую еженедельно; Prod – реальные данные с резервным копированием. |

**Особенности Stage окружения:**
- Полная копия production-конфигурации.
- Используется для нагрузочного тестирования перед релизом.
- Внешние API — реальные (с отдельными ключами) для проверки интеграций.
- Blue-Green деплой для тестирования новой версии с частью трафика.

**Вывод по разделу.** Многоуровневая архитектура окружений с изоляцией через namespace Kubernetes обеспечивает повторяемость и безопасность. Наиболее важным решением стало использование Infrastructure as Code (Terraform + Helm) для всех окружений — это гарантирует, что Stage максимально близок к Prod и снижает риск «сюрпризов» при релизе. Dev-окружение с синтетическими данными позволяет разработчикам быстро экспериментировать без риска повредить реальные данные.

### 7.3. Общие принципы CI/CD

**CI/CD Pipeline**

**1. Этапы сборки и проверки (CI):**

Каждый коммит в репозиторий запускает конвейер, включающий последовательные этапы:
- Сборка проекта
- Запуск unit-тестов
- Запуск интеграционных тестов
- Сборка контейнера Docker

**2. Quality Gates (ворота качества):**

Для прохождения в следующий этап код должен удовлетворять следующим критериям:
- Тестовое покрытие не менее 80%
- Все unit-тесты успешно пройдены
- Статический анализ кода (SonarQube) не содержит критических замечаний
- Сканирование уязвимостей (Trivy / Snyk) не выявляет критических CVE-уязвимостей

**3. Процесс развертывания по окружениям:**

| Окружение | Способ развертывания | Действия |
|-----------|---------------------|----------|
| **Dev** | Автоматически | Развертывание после успешного прохождения Quality Gates |
| **Test** | Автоматически | Развертывание после Dev; запуск полного набора автотестов |
| **Stage** | Вручную (с утверждением) | Нагрузочное тестирование (k6 / JMeter); User Acceptance Testing (UAT); утверждение от Tech Lead и Product Owner |
| **Prod** | Вручную (с утверждением) | Развертывание по стратегии Canary или Blue-Green; мониторинг ошибок и метрик; автоматический откат при превышении порога ошибок |

**4. Стратегии развертывания:**

| Стратегия | Применение | Описание |
|-----------|------------|----------|
| **Canary** | Основная стратегия для всех сервисов | Постепенное переключение трафика: 5% > 25% > 50% > 100%; возможность отката на любом этапе |
| **Blue-Green** | Для критических сервисов (Route Optimizer) | Две идентичные среды; мгновенный откат переключением трафика |
| **Feature Flags** | Для экспериментальных функций | Отделение деплоя от релиза; A/B-тестирование |

**5. Роль автоматизированного тестирования:**

| Уровень | Инструменты | Объект тестирования |
|---------|-------------|---------------------|
| Unit-тесты | JUnit / pytest | Отдельные компоненты и алгоритмы |
| Интеграционные тесты | Testcontainers | Взаимодействие с БД, Kafka, внешними API (с использованием моков) |
| API-тесты | Postman / Newman / REST Assured | Контракты API, валидация запросов и ответов |
| Нагрузочные тесты | k6 / JMeter | Проверка соответствия времени расчёта заданной асимптотической модели для n = 10, 50, 100, 200, 500 |
| E2E-тесты | Cypress / Playwright | Полные бизнес-сценарии через пользовательский интерфейс |

**6. Метрики успешности релиза:**

- Время развертывания: не более 15 минут
- Частота релизов: ежедневно (Dev/Test), еженедельно (Stage), ежемесячно (Prod — для мажорных релизов)
- Среднее время восстановления (MTTR): не более 15 минут
- Процент неудачных релизов: менее 5%


**Вывод по разделу.** CI/CD pipeline — это не просто инструмент автоматизации, а ключевой элемент обеспечения качества и скорости поставки. Наиболее важным решением стало использование Canary-деплоев с автоматическим откатом на основе метрик — это позволяет выпускать новые версии с минимальным риском. Feature Flags дают возможность выкатывать функции постепенно и тестировать их на реальных пользователях. Quality Gates на каждом этапе предотвращают попадание дефектного кода в production.

## Заключение

В рамках архитектурного проекта компонента «Оптимизация маршрутов доставки» системы TransLogix разработана комплексная архитектура на основе микросервисного подхода, событийно-ориентированной модели и современных облачных технологий. Архитектура включает 8 микросервисов с чёткими границами ответственности, определёнными на основе Domain-Driven Design, использует паттерны Saga и Eventual Consistency для обеспечения согласованности данных, а также реализует полный набор качественных атрибутов — масштабируемость, надёжность, безопасность и наблюдаемость.

Проект включет в себя функциональные требования по автоматическому построению и динамической корректировке маршрутов, интеграции с внешними сервисами, а также нефункциональные требования по производительности и доступности.

Реализация предложенной архитектуры позволит компании TransLogix повысить эффективность логистических операций, сократить время доставки и расход топлива, а также обеспечить гибкость и масштабируемость системы для будущего роста.
